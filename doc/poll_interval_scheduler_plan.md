# Plan: per-control интервалы опроса

Branch: `feature/SOFT-6492-poll-interval`

## Связанные планы

Часть 1 из 2. Часть 2 — `doc/poll_interval_dt8_split_plan.md`
(снятие bus-блокировки при опросе DT-8 цвета). Запускается после
завершения этой части.

## Архитектура опроса

Расписание опроса распределено по устройствам. Центрального
полл-планировщика с глобальной приоритезированной очередью нет —
ApplicationController работает как тонкий оркестратор, набирающий
за один tick **до 3 DALI-команд суммарно**, возможно, с разных
устройств.

- Каждое DALI-устройство **само** ведёт расписание дедлайнов для
  своих readable-контролов. Для каждого контрола используется либо
  собственный hardcoded-интервал (атрибут на `MqttControlBase`),
  либо fallback на bus-default (`buses[*].polling_interval`).
- **Раунд устройства = снимок** созревших контролов, зафиксированный
  в момент старта раунда. Контролы, ставшие due **после** старта
  снимка (включая опрошенные в этом же раунде на предыдущих
  tick'ах), в текущий снимок не попадают — ждут следующего раунда
  этого устройства.
- На каждом polling-tick'е ApplicationController набирает работу со
  всех устройств, у которых есть активный снимок или созревшие
  контролы, общим объёмом ≤ 3 DALI-команды:
  1. Для каждого рассматриваемого устройства:
     - если активен раунд с непустым остатком снимка — берёт
       очередную позицию (или несколько, если бюджет позволяет);
     - иначе — пытается начать новый раунд (срез созревших на
       `now`); если срез пуст, устройство в этом tick'е пропускается.
     Команды могут быть распределены между разными устройствами;
     конкретное распределение и порядок обхода — на усмотрение
     имплементатора, ограничение — суммарный бюджет ≤ 3 на tick.
  2. Когда первая взятая позиция — DT-8 цветовое чтение (~11
     команд), она одна занимает весь tick (см. ниже); другие
     устройства в этом tick'е не обслуживаются.
  3. Шаги попавших в tick устройств выполняются совместно
     (`asyncio.gather(..., return_exceptions=True)`); ошибка одного
     устройства не отменяет шаги других.
  4. AppController ждёт завершения всех корутин tick'а и публикует
     результаты в MQTT через существующий device-publisher;
     group-state-обвязка (закрепление source / publish ошибок в
     virtual-group device) — там же, где и сейчас.
  5. Когда у устройства снимок исчерпан → раунд этого устройства
     завершён; следующий раунд этого устройства начнётся, как только
     у него снова появятся созревшие контролы.
- Если у всех устройств снимки исчерпаны или пусты — polling-loop
  спит до минимума «time until next due» по всем устройствам, либо
  до прихода события в очередь задач (`EXECUTE_CONTROL`,
  commissioning, apply settings и т.п.).

Атомарность DT-8 цветового чтения: цветовое чтение — это один
логический контрол с одним дедлайном; в снимке оно занимает одну
позицию. Когда устройство берёт его на следующем шаге, корутина
возвращает план со стоимостью = реальное число команд (~11 для
RGBWAF). ApplicationController принимает план **даже если стоимость
больше бюджета 3** — на этом tick'е tick тратится целиком на
цветовое чтение, EXECUTE_CONTROL обрабатывается после его
завершения. Это известное ограничение Части 1, закрывается Частью 2.

## Сценарии

### 1. Контрол с захардкоженным собственным интервалом

В коде определения readable-контрола (`dali_controls.py`,
`dali_type21_parameters.py` и т.п.) задаётся явный интервал в
секундах. Контрол опрашивается с этим периодом независимо от
других контролов того же устройства.

Хранится как атрибут на `MqttControlBase`; значение `None` =
«использовать общешинный по умолчанию». В конфиг и UI не выносится.

Начальный набор хардкод-интервалов:

| Контрол | Интервал |
|---|---|
| `error_status` (база) | 120 с |
| `thermal_lamp_protection` (Type 21) | 120 с |
| `thermal_gear_protection` (Type 16) | 120 с |
| Все остальные readable-контролы | без override |

### 2. Контрол без собственного интервала

Опрашивается с интервалом из `buses[*].polling_interval`
(`/etc/wb-mqtt-dali.conf`). Конфиг пользователя не меняется,
у поля уточняется смысл (см. Documentation).

### 3. Бюджет 3 команды на один polling-tick

За один polling-tick AppController отправляет на шину суммарно не
более **3 DALI-команд** (исключение — DT-8 цветовое чтение, см.
сценарий 4). Эти 3 команды могут быть распределены между разными
устройствами шины; распределение — на усмотрение имплементатора.

Между tick'ами polling-loop возвращается к чтению очереди задач,
поэтому `EXECUTE_CONTROL` (`/on`) попадает между tick'ами. При
типичных DALI-таймингах (~45 мс на query+response) bus-серийная
длина tick'а ≤ ~135 мс независимо от того, на скольких устройствах
эти 3 команды выполняются, что даёт обработку `EXECUTE_CONTROL` не
реже раза в ~150 мс — за исключением tick'ов, целиком ушедших на
DT-8 цветовое чтение (см. сценарий 4).

### 4. DT-8 цветовой опрос — один шаг с превышением бюджета

В рамках Части 1 цветовое чтение DT-8 устройства реализовано
текущим способом (type-8 код выполняет всю последовательность
подзапросов под общей bus-блокировкой). С точки зрения архитектуры
устройства это один логический контрол, занимающий одну позицию в
снимке раунда. Когда устройство обслуживает эту позицию,
возвращается план со стоимостью = реальное число команд цветового
чтения (~11 для RGBWAF). ApplicationController принимает план даже
при превышении бюджета 3.

`EXECUTE_CONTROL` во время такого tick'а ждёт завершения цветового
чтения (до ~400 мс на RGBWAF). Это известное ограничение Части 1,
закрывается Частью 2.

### 5. Поведение при ошибке опроса

- Ошибка опроса однокомандного контрола (`r`/таймаут/framing) —
  публикуется как сейчас. Следующий дедлайн контрола =
  `previous_due + interval`, с clamp на `now + interval` если
  `previous_due + interval` уже в прошлом. Backoff не вводится.
- Ошибка для DT-8 цветовой записи — публикация ошибочного значения
  для всех её `ControlPollResult`-ов; следующий дедлайн как для
  однокомандного.
- В обоих случаях текущий шаг раунда считается выполненным: соотв.
  позиция вычеркивается из снимка раунда.

### 6. Что не меняется

- Очерёдность обработки `EXECUTE_CONTROL` относительно опроса
  (приоритет очереди задач).
- Поведение init/retry-init шага (alternation в polling-loop).
- Quiescent-режим: опрос приостановлен, расписания устройств
  заморожены.
- Group-state-контролы и публикация в виртуальные группы.
- После `EXECUTE_CONTROL` нет forced-poll контрола.
- Драйверная семантика для commissioning / apply settings.
- Сам code path type-8 цветового опроса (изменяется в Части 2).

## Краевые случаи

### КС-1. Опрошенный контрол снова созрел mid-раунда

Внутри раунда поллинг идёт по **снимку**, зафиксированному на старте
раунда устройства. Контролы, у которых `next_due` стал в прошлом
после старта снимка (включая только что опрошенные в текущем раунде
с коротким интервалом), в текущий снимок не попадают — они дождутся
следующего раунда этого устройства. Это исключает ситуацию, когда
быстрый контрол монополизирует свою долю polling-tick'а.

### КС-2. Apply-задача попадает в середину раунда

Apply-задачи (`_apply_*_task`) проходят через ту же очередь, что и
polling-tick'и; они дожидаются завершения текущего tick'а. После
любого успешного apply (per-device, group, bus):

- все per-control дедлайны затронутого устройства сбрасываются
  в `now` (созревают сразу);
- текущий раунд устройства, если был активен, **инвалидируется** —
  снимок выбрасывается.

На следующем tick'е у этого устройства начинается новый раунд с
новым снимком.

### КС-3. Удаление устройства в середине раунда

Задачи commissioning / reset / republish аналогично проходят через
очередь и дожидаются текущего tick'а. После удаления устройство и
его дедлайны исчезают вместе; если был активен раунд — он
выбрасывается. На следующих tick'ах ApplicationController мимо
удалённого устройства не идёт.

### КС-4. Quiescent во время раунда

Текущий tick доезжает до конца. Полл-loop, увидев флаг quiescent,
не запускает следующий tick. Активные раунды всех устройств
считаются инвалидированными (снимки не сохраняются). После выхода
из quiescent опрос продолжается обычным порядком, без burst и без
накопленных «долгов» — сработает clamp на `now + interval` для
просроченных дедлайнов.

### КС-5. set_polling_interval шины во время работы

Изменение `buses[*].polling_interval` через
`ApplicationController.set_polling_interval` (или эквивалент)
применяется сразу: для каждого inheriting-контрола (без
hardcoded-override) `next_due` сбрасывается в `now`. Контролы с
override не трогаются. У всех устройств активные раунды
инвалидируются (как при apply, КС-2).

### КС-6. Несколько контролов одного устройства созрели одновременно

Все попадают в снимок одного раунда устройства. Если других
устройств с работой нет, устройство получает все 3 слота tick'а:
снимок обслуживается за `ceil(N / 3)` tick'ов однокомандных
подходов плюс отдельный tick для DT-8 цветового чтения, если оно
тоже в снимке. Если у других устройств тоже есть созревшие позиции,
бюджет ≤ 3 на tick делится между ними — обслуживание этого снимка
занимает больше tick'ов, но бюджет латентности EXECUTE_CONTROL
остаётся прежним. Между tick'ами polling-loop возвращается к
очереди (`EXECUTE_CONTROL` пробивается).

### КС-7. Бюджет уже исчерпан в текущем tick'е

Бюджет ≤ 3 — суммарный на tick по всем устройствам. Когда tick
набрал 3 команды (с одного устройства или с нескольких),
дополнительные позиции из снимков ждут следующего tick'а — вне
зависимости от того, у того же устройства они или у других.

### КС-8. Ошибка корутины опроса

Корутина шага раунда может выбросить исключение (transmission
error, exhausted retries и т.п.). Обрабатывается так же, как
сегодня: публикация ошибочного значения для затронутых
`ControlPollResult`-ов + закрепление source в group state при
device-wide отказе. Snapshot обновляется как при успешном шаге
(позиция вычеркивается, дедлайны сдвигаются как в сценарии 5).
Шаги нескольких устройств одного tick'а гонятся через
`asyncio.gather(..., return_exceptions=True)`: ошибка одного
устройства не отменяет шаги других, каждое исключение
обрабатывается независимо.

### КС-9. Init устройства параллельно с polling-loop

Init по-прежнему обрабатывается отдельно от polling-tick'а
(alternation в `_polling_loop`, как сейчас). После завершения init
все readable-контролы свежеинициализированного устройства получают
`next_due = now`. На следующем polling-tick'е у этого устройства
стартует новый раунд.

## API

Новых RPC и публичных функций нет. Изменения видны только через:

| Поверхность | Что меняется |
|---|---|
| `/etc/wb-mqtt-dali.conf` (`buses[*].polling_interval`) | Имя/тип не меняются. Уточняется смысл: «интервал опроса по умолчанию для одного контрола без override» вместо «длительность полного круга по устройствам». Миграции конфига не нужны. |
| MQTT (топики `/devices/.../controls/...`) | Без новых топиков и meta-полей. Темп публикации значений readable-контролов меняется по новому расписанию. |
| AsyncAPI / JSON-RPC | Без изменений. |

Внутренний интерфейс между ApplicationController и устройством
(метод запроса плана шага раунда, метод старта/инвалидации
раунда, метод запроса «времени до ближайшего дедлайна») — на
усмотрение имплементатора, фиксируется тестами. Соглашение об
именах: новые методы не должны переименовывать существующие
идентификаторы (правило проекта); расширяющая сигнатура
существующего `poll_controls` допустима, новый отдельный метод
тоже допустим — выбор за имплементатором.

## Tests

`tests/`, поверх существующих фейков. Названия — ориентир.

- `test_control_with_explicit_interval_polled_at_its_rate` —
  контрол с интервалом 10 c опрашивается раз в ~10 c независимо от
  других readable-контролов того же устройства с дефолтом 5 c.
- `test_control_without_interval_falls_back_to_bus_default` —
  смена `polling_interval` шины меняет темп контролов без override;
  контрол с override сохраняет свой темп.
- `test_alarm_controls_use_120s_interval` — readable-контролы
  `error_status`, `thermal_lamp_protection`,
  `thermal_gear_protection` опрашиваются раз в 120 с независимо от
  `polling_interval` шины.
- `test_first_poll_after_init_happens_immediately` — после init все
  readable-контролы устройства опрашиваются в течение одного
  раунда этого устройства, при ≤ 3 cmd суммарно на tick.
- `test_polling_tick_budget_capped_at_three_commands` — суммарное
  число DALI-команд за один polling-tick ≤ 3 вне зависимости от
  числа задействованных устройств (исключение — DT-8 цветовая
  запись).
- `test_round_snapshot_excludes_controls_matured_mid_round` —
  контрол A с коротким интервалом, контрол B с длинным; A опрошен
  на первом tick'е раунда, к моменту следующего tick'а его
  `next_due` уже в прошлом, но он не попадает обратно в текущий
  снимок — повторно будет опрошен только в следующем раунде того
  же устройства.
- `test_tick_can_mix_commands_from_several_devices` — если у
  нескольких устройств есть созревшие контролы, в одном tick'е
  команды могут быть распределены между ними (≤ 3 команды
  суммарно); между устройствами никакого round-robin'а или
  cursor'а нет.
- `test_dt8_colour_step_consumes_whole_tick` — DT-8 цветовая
  запись в снимке порождает один tick с `send_commands` >3 cmd;
  AppController принимает превышение, другие устройства в этом
  tick'е не обслуживаются; снимок продолжается на следующем
  tick'е (если в нём остались другие позиции) или завершается.
- `test_apply_resets_deadlines_and_invalidates_round` — apply на
  устройстве в середине раунда: все per-control дедлайны
  устройства = `now`, активный раунд выброшен; на следующем
  polling-tick'е у устройства стартует новый раунд.
- `test_set_polling_interval_resets_inheriting_controls` — смена
  `polling_interval` шины: `next_due` контролов без override = `now`;
  контролы с override не тронуты.
- `test_poll_error_does_not_change_schedule` — ошибка опроса
  однокомандного контрола: позиция вычеркнута из снимка, следующий
  дедлайн = `previous_due + interval` (или clamp на `now + interval`,
  если первое в прошлом).
- `test_quiescent_mode_suspends_polling` — в quiescent-режиме
  polling-loop не запускает шаги раундов; активный раунд
  инвалидируется.
- `test_overdue_deadline_after_quiescent_does_not_burst_catch_up` —
  после длинного quiescent просроченные дедлайны опрашиваются по
  обычному раунду (≤ 3 cmd на tick), без буста накопленных
  пропусков.
- `test_device_removed_drops_its_polling_state` — удаление
  устройства: его дедлайны и активный раунд (если был) исчезают;
  на следующих tick'ах ApplicationController мимо удалённого
  устройства не идёт.
- `test_polling_loop_sleeps_until_earliest_deadline` — когда нет
  созревших контролов ни на одном устройстве, polling-loop спит до
  минимума «time until next due» по всем устройствам.

## Out of scope

- Разбиение DT-8 цветового опроса на отдельные шаги — Часть 2.
- Снятие bus-блокировки на время цветового чтения — Часть 2.
- Цель «`EXECUTE_CONTROL` ≤ 150 мс» на DT-8 шинах — Часть 2.
- Центральный полл-планировщик с глобальной приоритезированной
  очередью контролов — отдельной сущности нет, расписание
  распределено по устройствам, ApplicationController за tick
  фан-аутится по устройствам без сквозной приоритезации.
- Конфигурируемость per-control интервалов через
  `wb-mqtt-dali.conf` или RPC.
- Адаптация интервала под нагрузку шины, ошибки контрола, backoff.
- Изменение приоритизации `EXECUTE_CONTROL` относительно других
  задач очереди (commissioning, apply settings).
- Группировка по бус-времени (мс) или адаптивные пороги.
- Изменения в group publishing pipeline и virtual broadcast.
- Forced-poll контрола сразу после `EXECUTE_CONTROL`.
