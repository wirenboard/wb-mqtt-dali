# Plan: pre-emptable DT-8 цветовой опрос

Branch: `feature/SOFT-6492-dt8-poll-split` (предлагаемое имя; родитель
по веткам — после мерджа Части 1).

## Связанные планы

Часть 2 из 2. Опирается на инфраструктуру из
`doc/poll_interval_scheduler_plan.md` (per-control расписание, лимит
≤ 3 cmd на пачку, cross-device-микширование). До завершения Части 1
эта работа не имеет смысла.

## Сценарии

### 1. DT-8 цветовое чтение — несколько подзапросов планировщика

Чтение цвета DT-8 устройства состоит из нескольких подзапросов:
первая «опорная» пачка читает текущий уровень и активный
colour-type; дальше по одному запросу на каждый компонент:

| Подтип | Кол-во компонент-пачек | Размер компонент-пачки |
|---|---|---|
| RGBWAF | 4 (R, G, B, W) | 2 команды (DTR0 + QueryColourValue) |
| XY | 2 (X, Y) | 3 команды (DTR0 + QueryColourValue + QueryContentDTR0) |
| TC | 1 | 3 команды |
| PrimaryN | до 6 | 3 команды |

Каждый подзапрос едет в драйвер отдельным `send_commands`
(≤ 3 команд каждый, лимит из Части 1 соблюдается). Между
подзапросами polling-loop возвращается к очереди задач —
драйверная bus-блокировка на время всего цветового чтения **не**
удерживается.

### 2. Ответственность за промежуточное состояние

Промежуточное состояние одного незавершённого цветового чтения
(обнаруженный colour_type, частично прочитанные компоненты)
принадлежит type-8 коду устройства — тому же модулю, который и
сегодня собирает финальный комплект `ControlPollResult`-ов.

Планировщик ведёт только расписание дедлайнов и счёт «созревших»
контролов; в DALI-ответы и в частичные значения он не заглядывает.
Для планировщика интерфейс цветового опроса однообразен с
однокомандным контролом: «сделай следующий шаг» / «вот результат
для публикации» / «ничего не готово».

### 3. Ошибка отдельного подзапроса

При transmission error / framing / таймауте подзапрос ретраится
до 3 попыток в рамках того же цветового чтения — это поведение
текущего `query_colour_with_level`, оно сохраняется. Если 3
попытки исчерпаны, текущее цветовое чтение завершается публикацией
ошибочного значения для всех его `ControlPollResult`-ов; следующий
дедлайн = `now + interval`. Backoff не вводится.

### 4. EXECUTE_CONTROL во время цветового чтения

`/on`, пришедший в момент исполнения одного из подзапросов
цветового чтения, обрабатывается между подзапросами — до окончания
всего чтения. На шине из 4 RGBWAF DT-8 устройств при непрерывном
опросе `EXECUTE_CONTROL` подхватывается за **≤ 150 мс** (cost
query+response = 45 мс, forward-only = 30 мс).

### 5. Quiescent-режим и удаление устройства в середине чтения

- Вход в quiescent в середине цветового чтения: незавершённое
  чтение отбрасывается, partial state забывается. После выхода из
  режима чтение начинается с нуля по обычному расписанию.
- Удаление устройства (commissioning убрал, переинициализация и
  т.п.) в середине чтения: то же — partial state забывается, ни
  одного `ControlPollResult` не публикуется по этому чтению.

### 6. Что не меняется

- Поведение однокомандных контролов и расписание в целом.
- Интервалы (включая alarm-овеайды 120 с из Части 1).
- Лимит ≤ 3 cmd на пачку.
- Драйверная семантика для commissioning / apply settings —
  цепочки команд, которые сейчас идут одной транзакцией под единым
  bus-локом, остаются как есть.
- Список и состав публикуемых цветовых `ControlPollResult` —
  никаких новых/удалённых контролов.

## API

Новых RPC и публичных функций нет. Никаких изменений MQTT-топиков,
конфига или схем. Изменения только во внутренней реализации
type-8 цветового чтения и в способе, которым планировщик его
запускает.

## Tests

`tests/`, поверх существующих фейков. Названия — ориентир.

- `test_type8_colour_poll_split_into_subbatches` — для DT-8 RGBWAF
  цветовое чтение приходит в драйвер несколькими отдельными
  `send_commands` (первая опорная + по одной на компонент);
  результаты публикуются после последнего подзапроса.
- `test_type8_colour_poll_does_not_hold_bus_lock_across_subbatches` —
  между подзапросами цветового чтения других потребителей шины не
  блокирует (например, задача-«подсадная» из очереди исполняется
  между подзапросами на фейк-драйвере с лок-индикатором).
- `test_execute_control_preempts_between_poll_subbatches` — `/on`,
  пришедший во время цветового чтения, обрабатывается между
  подзапросами этого чтения, до его завершения.
- `test_execute_control_latency_under_150ms_in_4_rgbwaf_setup` —
  на шине из 4 RGBWAF DT-8 устройств с непрерывным опросом
  `EXECUTE_CONTROL` подхватывается за ≤ 150 мс по фейк-таймеру.
- `test_type8_subbatch_retries_up_to_three_times` — transient
  ошибка одной подпачки ретраится до 3 попыток в рамках того же
  цветового чтения.
- `test_type8_subbatch_failure_publishes_error_and_reschedules` —
  после исчерпания 3 попыток текущее чтение публикует ошибку для
  всех своих `ControlPollResult`-ов и перепланируется на
  `now + interval`.
- `test_type8_quiescent_mid_read_drops_partial_state` — вход в
  quiescent в середине цветового чтения отбрасывает partial state;
  после выхода ничего не догоняется бурно.
- `test_type8_device_removed_mid_read_drops_partial_state` —
  удаление устройства в середине чтения отбрасывает partial state,
  ни один `ControlPollResult` по этому чтению не публикуется.

## Out of scope

- Изменение списка и порядка цветовых компонентов.
- Изменение публикуемых `ControlPollResult`-контролов
  (`current_rgb`, `current_white`, `current_xy`, и т.д.).
- Изменение поведения при ошибках вне DT-8.
- Адаптивный backoff на ошибках type-8.
- Изменение драйверной семантики `run_sequence` для commissioning
  и apply settings.
- Forced-poll цветового контрола сразу после `EXECUTE_CONTROL`.
- Изменения интервалов из Части 1.
- Cross-device-микширование подпачек цветового чтения с командами
  однокомандных контролов в **один** `send_commands` — пакет
  отдельного цветового подзапроса целиком занимает свою отправку.
